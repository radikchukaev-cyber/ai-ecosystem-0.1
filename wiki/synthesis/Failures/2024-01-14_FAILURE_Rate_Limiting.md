# 💥 POSTMORTEM: ОШИБКА RATE LIMITING (2024-01-14)

> [!WARNING]
> **СИМПТОМ**
> Скрипт массового парсинга YouTube API внезапно упал с кодом 403 (Quota Exceeded) и прервал всю цепочку задач Шефа.
> Система не смогла восстановиться самостоятельно и потребовала ручного вмешательства.

## 1. АНАЛИЗ (ROOT CAUSE) - 5 WHYS
1. **Почему скрипт упал?**
   Потому что YouTube API вернул ошибку превышения квоты.
2. **Почему квота была превышена?**
   Потому что скрипт VULCAN отправлял 100 запросов в секунду (параллельно), исчерпав дневной лимит за 5 минут.
3. **Почему скрипт отправлял так много запросов?**
   Потому что в коде использовался `asyncio.gather` без ограничения семафором (Semaphore).
4. **Почему не было ограничения скорости?**
   Потому что в `wiki/concepts/Code_Patterns.md` не было стандарта по Rate Limiting для API.
5. **Почему не было fallback-механизма?**
   Система не умела делать Exponential Backoff и Circuit Breaker, скрипт просто панически завершался.

**Корневая причина:** Отсутствие контроля параллелизма (concurrency) и обработки ошибок API на уровне архитектуры.

## 2. ПОСЛЕДСТВИЯ
- Потеряно 3 часа рабочего времени Шефа.
- Задачи из `MEMORY.md` зависли в статусе `[IN_PROGRESS]`, что сломало логику RAMS.

## 3. ПРИНЯТЫЕ МЕРЫ (ACTION ITEMS)
- **VULCAN:** Добавил `asyncio.Semaphore(5)` во все сетевые скрипты.
- **SOCRATES:** Создал новый концепт: `wiki/concepts/API_Error_Handling.md`.
- **RAMS:** Инструкции обновлены: теперь любой API-скрипт обязан иметь retry-логику.
- **КАСПЕР:** Добавил скрипт автоматического снятия зависших `[IN_PROGRESS]` задач через 1 час бездействия.

## 4. ВЫВОД ДЛЯ СИСТЕМЫ
Скорость исполнения не должна угрожать стабильности. Лучше парсить данные 10 минут надежно, чем за 10 секунд исчерпать квоту API на сутки.
