🔧 КАК ИСПОЛЬЗОВАТЬ (ПОШАГОВО):

ЭТАП 1: ПОДГОТОВКА (Перед кодингом)
┌─ Прочитаю спецификацию (что писать?)
├─ Ищу похожее в [[wiki/synthesis/History/]] (было ли раньше?)
├─ Проверяю [[wiki/concepts/Code_Patterns.md]] (готовый шаблон?)
├─ Считаю сложность (часов, токенов)
├─ Готовлю тестовые данные
└─ Открываю текстовый редактор

ЭТАП 2: СТРУКТУРА КОДА
┌─ Разбиваю на функции (максимум 50 строк каждая)
├─ Каждой функции даю понятное имя (не func_1, не helper)
├─ Комментарии на РУССКОМ
│  └─ # Парсим JSON из YouTube API
│     # Проверяем есть ли ошибки
│     # Если ошибка — логируем и пропускаем
├─ Константы в CAPS (API_TIMEOUT = 30)
└─ Функции в snake_case (parse_json, get_data)

ПРИМЕР СТРУКТУРЫ:
parse_youtube_api.py
├─ Импорты (все нужные библиотеки)
├─ Константы (API_KEY = ..., TIMEOUT = 30)
├─ Helper функции (get_channel, parse_video)
├─ Главная функция (main или run)
└─ Блок if name == "main": (как запускать)

ЭТАП 3: ОБРАБОТКА ОШИБОК (Защита от падения)
┌─ Каждый вызов API обёрнут в try-except
│  ├─ try: request_to_api()
│  ├─ except TimeoutError: log_error("timeout", url)
│  ├─ except JSONDecodeError: log_error("json", response)
│  └─ except Exception as e: log_error("unknown", str(e))
│
├─ Валидирую входные данные
│  ├─ if not url: raise ValueError("URL is empty")
│  ├─ if not isinstance(data, dict): raise TypeError("Expected dict")
│  └─ if len(data) > MAX_SIZE: raise ValueError("Too large")
│
├─ Логирую каждый шаг (не буду молчать)
│  ├─ logger.info(f"Processing {url}")
│  ├─ logger.debug(f"Response: {response}")
│  └─ logger.error(f"Failed: {error}")
│
└─ Возвращаю результат или None (ничего не крушу)

ЭТАП 4: ТЕСТИРУЕМОСТЬ (Пишу так, чтобы можно было проверить)
┌─ Функции чистые (input → output, без side effects)
├─ Зависимости передаю параметром (не глобальное состояние)
├─ Готовлю моки для тестов
│  ├─ mock_api_response = {"status": "ok", "data": [...]}
│  └─ Использую mock вместо реального API в тестах
│
└─ Документирую входы-выходы
   ├─ def parse_json(raw_data: str) -> dict:
   │      """
   │      Парсит JSON строку в словарь
   │      Args: raw_data (str) — сырой JSON
   │      Returns: dict или пустой dict если ошибка
   │      Raises: None (обработаны все ошибки)
   │      """

ЭТАП 5: ПРОИЗВОДИТЕЛЬНОСТЬ (Код работает быстро)
┌─ Не делаю O(n²) если можно O(n)
├─ Кеширую результаты если нужно
├─ Не загружаю в память весь файл (5GB? читаю по частям)
├─ Параллелизирую где можно (threading, asyncio)
└─ Профилирую (где узкое место?)

ЭТАП 6: ДОКУМЕНТАЦИЯ (Скоро забуду что писал)
┌─ Файл README.md в папке
│  └─ Что это делает? Как запускать? Примеры?
├─ Docstrings в каждой функции
│  └─ """Что это, входы, выходы, примеры"""
├─ Комментарии в сложных местах
│  └─ # Вычисляем хэш для дедупликации
└─ Пример использования в конце файла
   ├─ if __name__ == "__main__":
   │      result = parse_json(example_data)

ПРИМЕР 1: ПАРСЕР JSON (С ОБРАБОТКОЙ ОШИБОК)

```python
import json
import logging

# Настройка логирования
logger = logging.getLogger(__name__)

def parse_json_safe(raw_data: str) -> dict:
    """
    Безопасно парсит JSON строку.
    
    Args:
        raw_data (str): JSON строка
        
    Returns:
        dict: Распарсенные данные или {}
        
    Example:
        >>> data = '{"name": "test"}'
        >>> parse_json_safe(data)
        {'name': 'test'}
    """
    # Проверяем что это строка
    if not isinstance(raw_data, str):
        logger.error(f"Expected string, got {type(raw_data)}")
        return {}
    
    # Проверяем что не пусто
    if not raw_data.strip():
        logger.warning("Empty data provided")
        return {}
    
    # Пытаемся парсить
    try:
        result = json.loads(raw_data)
        logger.info(f"Parsed JSON with {len(result)} keys")
        return result
        
    except json.JSONDecodeError as e:
        logger.error(f"Invalid JSON: {e}")
        return {}
        
    except Exception as e:
        logger.error(f"Unexpected error: {type(e).__name__}: {e}")
        return {}

if __name__ == "__main__":
    # Тестируем
    test_data = '{"name": "test", "value": 123}'
    result = parse_json_safe(test_data)
    print(f"Result: {result}")
```

ПРИМЕР 2: ЗАПРОС К API (С RETRY)

```python
import requests
import time
from typing import Optional

MAX_RETRIES = 3
TIMEOUT = 10
BACKOFF_FACTOR = 2  # 1сек, 2сек, 4сек

def get_data_from_api(url: str, timeout: int = TIMEOUT) -> Optional[dict]:
    """
    Получает данные с API с автоматическими повторами.
    
    Args:
        url (str): URL API
        timeout (int): Таймаут в секундах
        
    Returns:
        dict или None
    """
    logger.info(f"Requesting {url}")
    
    for attempt in range(1, MAX_RETRIES + 1):
        try:
            response = requests.get(url, timeout=timeout)
            
            # Проверяем статус
            if response.status_code == 200:
                logger.info(f"Success on attempt {attempt}")
                return response.json()
            else:
                logger.warning(f"Status {response.status_code} on attempt {attempt}")
                
        except requests.Timeout:
            logger.warning(f"Timeout on attempt {attempt}")
            
        except requests.ConnectionError:
            logger.warning(f"Connection error on attempt {attempt}")
            
        except Exception as e:
            logger.error(f"Unexpected error: {e}")
            return None
        
        # Ждём перед следующей попыткой
        if attempt < MAX_RETRIES:
            wait_time = BACKOFF_FACTOR ** (attempt - 1)
            logger.info(f"Retrying in {wait_time} seconds...")
            time.sleep(wait_time)
    
    logger.error(f"Failed after {MAX_RETRIES} attempts")
    return None
```

ЭТАП 2: ТРИ ТИПА ТЕСТОВ

ТИП 1: UNIT ТЕСТЫ (тестирую одну функцию)
┌─ Что: парсит ли parse_json корректный JSON?
├─ Входные данные: '{"name": "test"}'
├─ Ожидаемый результат: {"name": "test"}
└─ Проверяю: result == {"name": "test"}

```python
def test_parse_valid_json():
    """Тестирует парсинг корректного JSON"""
    input_data = '{"name": "test", "age": 25}'
    expected = {"name": "test", "age": 25}
    
    result = parse_json_safe(input_data)
    
    assert result == expected
```

ТИП 2: EDGE CASE ТЕСТЫ (странные входы)
┌─ Пустая строка: ''
├─ Null: None
├─ Огромный JSON: 10MB
├─ Спецсимволы: '\n\t\r'
├─ Поломанный JSON: '{"unclosed": '
└─ Типы: число вместо строки: 123

```python
def test_parse_empty_string():
    """Тестирует с пустой строкой"""
    result = parse_json_safe('')
    assert result == {}

def test_parse_none():
    """Тестирует с None"""
    result = parse_json_safe(None)
    assert result == {}

def test_parse_invalid_json():
    """Тестирует с поломанным JSON"""
    result = parse_json_safe('{"unclosed": ')
    assert result == {}

def test_parse_large_json():
    """Тестирует с большим JSON"""
    large_data = '{"' + ('key' * 10000) + '": "value"}'
    result = parse_json_safe(large_data)
    # Должно не упасть, должна вернуть результат или {}
    assert isinstance(result, dict)
```

ТИП 3: ИНТЕГРАЦИОННЫЕ ТЕСТЫ (тестирую несколько функций вместе)
┌─ Может ли parse_json + validate_data работать вместе?
├─ Может ли get_data_from_api + parse_json вместе работать?
└─ Может ли весь pipeline работать end-to-end?

```python
def test_api_to_parse_pipeline(mock_api):
    """Тестирует весь pipeline: API → парсинг → валидация"""
    # Мокируем API ответ
    mock_api.return_value = '{"users": [{"id": 1}]}'
    
    # Запускаем pipeline
    raw_data = mock_api()
    parsed = parse_json_safe(raw_data)
    validated = validate_data(parsed)
    
    # Проверяем результат
    assert validated is not None
    assert len(validated.get('users', [])) == 1
```

ЭТАП 3: МОКИ (Подделываю внешние зависимости)

Почему моки? Потому что:
- API может быть медленный
- API может быть платный
- API может быть недоступный
- Я хочу контролировать результат

```python
from unittest.mock import patch, Mock

def test_api_call_with_timeout(mock_requests):
    """Тестирует обработку таймаута"""
    # Подделываем что API бросит TimeoutError
    mock_requests.get.side_effect = TimeoutError()
    
    result = get_data_from_api("http://example.com")
    
    # Должно вернуть None (безопасно)
```

ЭТАП 4: ПАРАМЕТРИЗИРОВАННЫЕ ТЕСТЫ (один тест для много данных)

Вместо:
```python
def test_parse_1(): result = parse_json_safe('{}'); assert result == {}
def test_parse_2(): result = parse_json_safe('{"a":1}'); assert ...
def test_parse_3(): result = parse_json_safe('{"b":2}'); assert ...
```

Пишу:
```python
import pytest

@pytest.mark.parametrize("input_data,expected", [
    ('{}', {}),
    ('{"a": 1}', {"a": 1}),
    ('{"b": 2}', {"b": 2}),
    ('{"a": [1,2,3]}', {"a": [1, 2, 3]}),
])
def test_parse_various_json(input_data, expected):
    """Тестирует разные варианты JSON"""
    result = parse_json_safe(input_data)
    assert result == expected
```

```python
import pytest
from unittest.mock import patch, Mock
from code.parser import parse_json_safe, validate_data, get_data_from_api

# ===== UNIT ТЕСТЫ =====

class TestParseJsonSafe:
    """Тесты функции parse_json_safe"""
    
    def test_valid_json(self):
        """Валидный JSON парсится правильно"""
        data = '{"name": "test"}'
        result = parse_json_safe(data)
        assert result == {"name": "test"}
    
    def test_empty_string(self):
        """Пустая строка возвращает {}"""
        result = parse_json_safe('')
        assert result == {}
    
    def test_none_input(self):
        """None возвращает {}"""
        result = parse_json_safe(None)
        assert result == {}
    
    def test_invalid_json(self):
        """Поломанный JSON возвращает {}"""
        result = parse_json_safe('{"unclosed": ')
        assert result == {}
    
    @pytest.mark.parametrize("input_data,expected", [
        ('{}', {}),
        ('{"a": 1}', {"a": 1}),
        ('[]', []),  # Массив тоже валиден
    ])
    def test_various_json(self, input_data, expected):
        """Тестирует разные структуры"""
        result = parse_json_safe(input_data)
        assert result == expected

# ===== ИНТЕГРАЦИОННЫЕ ТЕСТЫ =====

class TestApiPipeline:
    """Тесты для API → парсинг pipeline"""
    
    @patch('requests.get')
    def test_api_success(self, mock_get):
        """API возвращает успех → парсим"""
        mock_get.return_value = Mock(status_code=200, json=lambda: {"data": "test"})
        
        result = get_data_from_api("http://example.com")
        
        assert result == {"data": "test"}
    
    @patch('requests.get')
    def test_api_timeout(self, mock_get):
        """API таймаут → возвращаем None"""
        import requests
        mock_get.side_effect = requests.Timeout()
        
        result = get_data_from_api("http://example.com")
        
        assert result is None
    
    @patch('requests.get')
    def test_api_retry_success(self, mock_get):
        """Первая попытка упадет, вторая сработает"""
        import requests
        mock_get.side_effect = [requests.Timeout(), Mock(status_code=200, json=lambda: {"ok": True})]
        
        result = get_data_from_api("http://example.com")
        
        assert result == {"ok": True}
        assert mock_get.call_count == 2  # Проверяем что было 2 попытки
```

ЭТАП 6: ИСПРАВЛЕНИЕ

После того как нашел причину, фиксу:

```python
# БЫЛО (падает):
def parse_json_safe(raw_data: str) -> dict:
    return json.loads(raw_data)

# СТАЛО (безопасно):
def parse_json_safe(raw_data: str) -> dict:
    if not isinstance(raw_data, str):
        logger.error(f"Expected str, got {type(raw_data)}")
        return {}
    
    if not raw_data.strip():
        logger.warning("Empty data")
        return {}
    
    try:
        return json.loads(raw_data)
    except json.JSONDecodeError as e:
        logger.error(f"JSON parse error at {e.pos}: {e.msg}")
        return {}
    except Exception as e:
        logger.error(f"Unexpected error: {e}")
        return {}
```

💡 ПРИМЕРЫ ДЕБАГИНГА:

ПРИМЕР 1: СКРЫТЫЙ БАГ (Код работает но результат неправильный)
Жалоба: "Количество видео неправильное, должно быть 100, а приходит 95"
Расследование:
├─ Логирую входные данные
│  └─ "Фетчил с API 100 видео"
├─ Логирую в середине
│  └─ "Распарсил 100 видео"
├─ Логирую в конце
│  └─ "После фильтра: 95 видео"
│     └─ ← Вот источник! Фильтр выбрасывает 5 видео!
Нашел причину:
filter_videos() имеет условие: if video.duration > 60  ← Выбрасывает короткие!
Исправление:
def filter_videos(videos, min_duration=None):
if min_duration is None:
return videos  # Без фильтра по умолчанию
return [v for v in videos if v.duration >= min_duration]

ПРИМЕР 2: RACE CONDITION (Многопоточный баг)
Жалоба: "Иногда работает, иногда падает. Не могу воспроизвести"
Расследование:
├─ Запускаю 100 раз
│  └─ Падает в 30% случаев (не детерминировано!)
├─ Подозреваю многопоточность
├─ Смотрю код:
│  ├─ Нить 1 читает: global_data['count'] = 5
│  └─ Нить 2 пишет: global_data['count'] += 1
│     └─ ← Race condition! Не знают друг о друге
Исправление:
from threading import Lock
lock = Lock()
with lock:
global_data['count'] += 1  # Теперь безопасно

ЭТАП 1: ПРЕ-DEPLOYMENT ЧЕКЛИСТ

До того как крутить кнопку deploy:

□ Все тесты зелёные?
  └─ pytest tests/ --cov=. (Coverage >= 80%)

□ Code review пройдён?
  └─ SOCRATES посмотрел код

□ Нет TODO в коде?
  └─ grep -r "TODO\|FIXME" code/ → ДОЛЖНО БЫТЬ ПУСТО

□ Переменные окружения готовы?
  ├─ API_KEY установлены?
  ├─ DATABASE_URL установлены?
  ├─ SECRET_KEY не в GitHub?
  └─ .env файл НЕ в гит!

□ Dependencies пинились?
  ├─ pip freeze > requirements.txt
  └─ Нет версий которые не работают вместе

□ Документация обновлена?
  ├─ README.md актуален?
  ├─ DEPLOYMENT.md существует?
  └─ Инструкции понятны новичку?

□ Мониторинг включен?
  ├─ Логи пишутся?
  ├─ Метрики собираются?
  └─ Алерты настроены?

□ План отката готов?
  ├─ Знаю как откатить?
  ├─ Знаю команду откатить?
  └─ Тестировал откат раньше?

ЭТАП 2: STAGING ENVIRONMENT (Копия production для тестов)

Перед production, деплоим в staging:
СТРУКТУРА:
├─ Development (мой компьютер)
│  └─ Все может быть сломано
│
├─ Staging (копия production)
│  └─ Тестирую код точно как в production
│
└─ Production (реальная система, работает для пользователей)
└─ Здесь нельзя ломать!

```bash
# Деплой в staging
git push origin main:staging  # пушу в ветку staging

# На staging сервере
git pull
python setup.py install
pytest tests/ -v
python app.py --port 8081  # Запускаю staging version

# Тестирую как пользователь
curl http://staging.example.com/api/test
# {"status": "ok"} ✅ Работает!

# Симулирую сбой (убиваю процесс)
kill -9 $(pgrep -f app.py)

# Проверяю что сработал rollback
curl http://staging.example.com/api/test
# {"status": "ok"} ✅ Восстановилась старая версия!
```

ЭТАП 3: DEPLOYMENT (Пускаю в production)

```bash
# Только если staging прошёл успешно!

# Способ 1: Blue-Green Deploy (два набора серверов)
# Blue (старый) ←→ Green (новый)
#
# 1. Поднимаю новую версию на Green серверах
# 2. Тестирую Green серверы
# 3. Переключаю трафик Blue → Green
# 4. Blue остается как откат если что-то упадет

# Способ 2: Canary Deploy (постепенный выпуск)
# 5% трафика → новый код
# Если ошибки → откатываю всех
# Если ОК → 50% трафика
# Если ОК → 100% трафика

# Способ 3: Rolling Deploy (постепенное обновление серверов)
# 1. Сервер 1: выключу, обновлю, включу
# 2. Сервер 2: выключу, обновлю, включу
# 3. Пока всегда есть живые серверы (0 downtime)
```

ЭТАП 4: КОМАНДА DEPLOY (Точно по чеклисту)

```bash
#!/bin/bash
# deploy.sh — скрипт развёртывания

set -e  # Упадёш если что-то не так

echo "[1/6] Checking tests..."
pytest tests/ --cov=. || exit 1

echo "[2/6] Checking for TODOs..."
if grep -r "TODO\|FIXME" code/; then
    echo "ERROR: Found TODOs in code!"
    exit 1
fi

echo "[3/6] Building..."
pip install -r requirements.txt
python setup.py build

echo "[4/6] Backing up current version..."
cp -r /app /app.backup.$(date +%s)

echo "[5/6] Deploying new version..."
cp -r build/* /app/

echo "[6/6] Running post-deploy tests..."
pytest tests/integration/ --live-api

echo "✅ Deployment successful!"
echo "Previous version backed up at: /app.backup.*"
```

ЭТАП 5: МОНИТОРИНГ (Смотрю что происходит)

После деплоя, жди 5 минут и смотри:

```python
# monitoring.py — что проверяю после деплоя

import requests
import logging

logger = logging.getLogger(__name__)

def check_health():
    """Проверяю что код работает"""
    try:
        response = requests.get("http://api.example.com/health", timeout=5)
        if response.status_code != 200:
            logger.error(f"Health check failed: {response.status_code}")
            return False
        logger.info("✅ Health check OK")
        return True
    except Exception as e:
        logger.error(f"Health check exception: {e}")
        return False

def check_error_rate():
    """Проверяю что ошибок не больше 1%"""
    total_requests = 10000
    error_requests = 50
    error_rate = error_requests / total_requests
    
    if error_rate > 0.01:  # > 1%
        logger.error(f"Error rate too high: {error_rate:.2%}")
        return False
    
    logger.info(f"✅ Error rate OK: {error_rate:.2%}")
    return True

def check_latency():
    """Проверяю что скорость не упала"""
    avg_latency = 150  # ms
    if avg_latency > 200:  # > 200ms = проблема
        logger.error(f"Latency too high: {avg_latency}ms")
        return False
    
    logger.info(f"✅ Latency OK: {avg_latency}ms")
    return True

# Запускаю все проверки
if __name__ == "__main__":
    checks = [
        check_health(),
        check_error_rate(),
        check_latency(),
    ]
    
    if all(checks):
        logger.info("✅ All checks passed! Deploy successful!")
    else:
        logger.error("❌ Some checks failed! Consider rollback!")
```

```python
import cProfile
import pstats
```

ЭТАП 2: ВЫЯВЛЕНИЕ ПРОБЛЕМЫ

Проблема 1: НЕЭФФЕКТИВНЫЙ АЛГОРИТМ
┌─ Было: O(n²) — для 1000 элементов: 1,000,000 операций
├─ Стало: O(n log n) — для 1000 элементов: 10,000 операций
└─ Улучшение: 100x быстрее!

```python
# ❌ МЕДЛЕННО: O(n²)
def find_duplicates_slow(data):
    duplicates = []
    for i in range(len(data)):
        for j in range(i+1, len(data)):
            if data[i] == data[j]:
                duplicates.append(data[i])
    return duplicates

# ✅ БЫСТРО: O(n)
def find_duplicates_fast(data):
    seen = set()
    duplicates = []
    for item in data:
        if item in seen:
            duplicates.append(item)
        seen.add(item)
    return duplicates

# Тестирую:
data = list(range(10000))
import time

start = time.time()
find_duplicates_slow(data)
slow_time = time.time() - start  # 5 секунд!

start = time.time()
find_duplicates_fast(data)
fast_time = time.time() - start  # 0.05 секунд!

print(f"Улучшение: {slow_time / fast_time:.1f}x быстрее!")  # 100x!
```

Проблема 2: ПОВТОРЯЮЩИЕСЯ ВЫЧИСЛЕНИЯ (Кеширование)
┌─ Было: Каждый раз считаю заново (дорого!)
├─ Стало: Кеширую результат (быстро!)
└─ Улучшение: зависит от частоты запросов

```python
# ❌ ДОРОГО: Каждый раз считаю заново
def expensive_calculation(n):
    """Вычисляю факториал (дорого)"""
    result = 1
    for i in range(1, n+1):
        result *= i
    return result

# ✅ ДЕШЕВО: Кеширую (используя functools.lru_cache)
from functools import lru_cache

@lru_cache(maxsize=128)  # Кешу последние 128 результатов
def expensive_calculation_cached(n):
    result = 1
    for i in range(1, n+1):
        result *= i
    return result

# Тестирую:
import time

# Первый вызов (медленно, считаю):
start = time.time()
expensive_calculation_cached(100)
print(f"First call: {time.time() - start:.4f}s")

# Второй вызов (быстро, из кеша):
start = time.time()
expensive_calculation_cached(100)
print(f"Second call: {time.time() - start:.6f}s")  # 1000x быстрее!
```

Проблема 3: СЛИШКОМ МНОГО В ПАМЯТИ
┌─ Было: Загружаю весь файл (5GB) в память
├─ Стало: Читаю по частям (1MB за раз)
└─ Улучшение: memory 5000x меньше!

```python
# ❌ МНОГО ПАМЯТИ: Загружаю всё сразу
def process_huge_file_bad(filename):
    with open(filename, 'r') as f:
        data = f.read()  # 5GB в памяти!
    for line in data.split('\n'):
        process_line(line)

# ✅ МАЛО ПАМЯТИ: Читаю по строкам
def process_huge_file_good(filename):
    with open(filename, 'r') as f:
        for line in f:  # Одна строка в памяти за раз
            process_line(line)

# Разница:
# Bad: 5000 MB памяти
# Good: 5 MB памяти (1000x меньше!)
```

Проблема 4: ПАРАЛЛЕЛИЗМ (Делаю несколько задач одновременно)
┌─ Было: Последовательно (1, потом 2, потом 3)
├─ Стало: Параллельно (1 и 2 и 3 вместе)
└─ Улучшение: 4x быстрее на 4-ядерном процессоре

```python
import concurrent.futures

# ❌ МЕДЛЕННО: Последовательно
def process_urls_slow(urls):
    results = []
    for url in urls:
        result = requests.get(url).json()
        results.append(result)
    return results

# ✅ БЫСТРО: Параллельно
def process_urls_fast(urls):
    with concurrent.futures.ThreadPoolExecutor(max_workers=4) as executor:
        futures = [executor.submit(requests.get, url) for url in urls]
        results = [f.result().json() for f in futures]
    return results

# На 100 URL'ов:
# Slow: 100 сек (100 × 1сек на URL)
# Fast: 25 сек (100 ÷ 4 параллельно)
# Улучшение: 4x!
```

ЭТАП 3: ОПТИМИЗАЦИЯ ТОКЕНОВ (Дешевый AI)

Проблема: API запрос стоит 1000 токенов, слишком дорого!

Решение 1: БАТЧИНГ (группировка запросов)
```python
# ❌ ДОРОГО: Отправляю 1 запрос за раз
for item in 1000_items:
    result = api.process(item)  # 1000 × 1000 = 1M токенов!

# ✅ ДЕШЕВО: Отправляю 100 запросов в одном
batch_size = 100
for i in range(0, len(items), batch_size):
    batch = items[i:i+batch_size]
    result = api.process_batch(batch)  # 10 × 2000 = 20K токенов!
# Экономия: 50x!
```

Решение 2: КЕШИРОВАНИЕ API (не спрашиваю дважды)

```python
from functools import lru_cache
```

КОМПРИМИРОВАННЫЙ (200 слов):
YouTube API
YouTube API — для работы с видео, каналами, комментариями.
Аутентификация: OAuth 2.0. Получить ключ в Google Cloud Console.
Основные endpoints:

videos.list — информация о видео (название, дата, статистика)
channels.list — инфо о канале (подписчики, видео)
comments.list — комментарии к видео
search.list — поиск видео

Rate Limits: 10,000 квот в день. Каждый запрос = 1-100 квот.
Примеры ошибок:

403: API не включен в проекте
401: некорректный ключ
429: превышен лимит квот

Цена: бесплатно до 10K квот/день, потом платно

Компрессия: 2000 слов → 200 слов (10x!)
